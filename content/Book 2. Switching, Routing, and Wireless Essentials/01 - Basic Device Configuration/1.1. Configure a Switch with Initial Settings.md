# 1.1 Configure a Switch with Initial Settings

---

## 1.1.1 Последовательность загрузки коммутатора

После включения Cisco switch проходит **пять этапов загрузки**.

### Шаг 1. POST

Сначала коммутатор запускает программу **POST (Power-On Self-Test)**, которая хранится в **ROM**.

POST проверяет:

- CPU;
- DRAM;
- часть flash-памяти, которая используется как файловая система.

Если проверка не проходит, нормальная загрузка IOS не продолжится.

### Шаг 2. Загрузка boot loader

Далее устройство загружает **boot loader**. Это небольшая программа, которая тоже хранится в ROM и запускается сразу после успешного завершения POST.

### Шаг 3. Низкоуровневая инициализация CPU

Boot loader выполняет базовую инициализацию процессора:

- подготавливает регистры CPU;
- определяет, как отображается физическая память;
- определяет объём памяти;
- определяет её скорость.

### Шаг 4. Инициализация flash file system

После этого boot loader подготавливает файловую систему flash-памяти на системной плате.

### Шаг 5. Загрузка IOS

В конце boot loader находит стандартный образ **IOS**, загружает его в память и передаёт управление операционной системе.

**Итог:** сначала проверяется железо, потом запускается boot loader, затем подготавливается память и только после этого загружается IOS.

## 1.1.2 Команда boot system

Коммутатор пытается загрузиться автоматически, используя информацию из переменной окружения **BOOT**.

Если переменная **BOOT** не задана, switch пытается найти и запустить первый подходящий исполняемый файл, который сможет обнаружить.

На коммутаторах **Catalyst 2960** файл образа IOS обычно находится в папке с тем же именем, что и сам файл образа без расширения `.bin`.

После загрузки IOS устройство инициализирует интерфейсы, используя команды Cisco IOS из файла **startup-config**.

Файл startup-config:

- называется `config.text`;
- находится во flash-памяти.

### Пример команды boot system

```bash
S1(config)# boot system flash:/c2960-lanbasek9-mz.150-2.SE/c2960-lanbasek9-mz.150-2.SE.bin
```

Эта команда в глобальном режиме конфигурации задаёт путь к IOS, который должен использоваться при следующей загрузке.

### Как проверить текущий boot file

Для просмотра текущего boot-файла используется команда:

```bash
show boot
```

**Важный момент:** если путь в BOOT указан неверно, коммутатор может не загрузить нужный IOS-образ автоматически.

## 1.1.3 LED-индикаторы коммутатора

У Cisco Catalyst switches есть несколько **LED-индикаторов состояния**, по которым можно быстро понять, что происходит с устройством.

У разных моделей:

- набор LED может отличаться;
- их расположение на передней панели тоже может отличаться.

На Cisco Catalyst 2960 обычно используются следующие индикаторы:

1. `SYST`
2. `RPS`
3. `STAT`
4. `DUPLX`
5. `SPEED`
6. `PoE`
7. кнопка `Mode`
8. LED-индикаторы портов

Кнопка **Mode** переключает режим отображения портовых индикаторов:

- статус порта;
- duplex;
- скорость;
- PoE, если поддерживается.

### SYST

**System LED** показывает, получает ли устройство питание и работает ли оно нормально.

- `off` — питание не подаётся;
- `green` — коммутатор работает нормально;
- `amber` — питание есть, но устройство работает некорректно.

### RPS

**RPS LED** показывает состояние **Redundant Power System**.

- `off` — RPS выключен или подключён неправильно;
- `green` — RPS подключён и готов дать резервное питание;
- `blinking green` — RPS подключён, но сейчас занят питанием другого устройства;
- `amber` — RPS в standby или в состоянии ошибки;
- `blinking amber` — встроенный блок питания switch отказал, и питание подаётся через RPS.

### STAT

**Port Status LED** показывает состояние портов.

- `off` — линка нет или порт administratively down;
- `green` — линк есть;
- `blinking green` — идёт передача или приём данных;
- `alternating green-amber` — ошибка линка;
- `amber` — порт заблокирован, чтобы не возникла петля в forwarding domain;
- `blinking amber` — порт заблокирован для предотвращения возможной петли.

Обычно после активации порта состояние `amber` может наблюдаться первые примерно **30 секунд**.

### DUPLX

**Port Duplex LED** показывает режим duplex.

- если LED порта `off` — порт работает в **half-duplex**;
- если LED порта `green` — порт работает в **full-duplex**.

### SPEED

**Port Speed LED** показывает скорость порта.

- `off` — `10 Mbps`;
- `green` — `100 Mbps`;
- `blinking green` — `1000 Mbps`.

### PoE

Если коммутатор поддерживает **Power over Ethernet**, у него есть **PoE Mode LED**.

- `off` — режим PoE не выбран, и ни один порт не находится в ошибке PoE;
- `blinking amber` — режим PoE не выбран, но как минимум на одном порту есть отказ питания или PoE fault;
- `green` — выбран режим PoE, и теперь состояние портов отображает именно питание.

Если выбран режим PoE, то LED порта означает:

- `off` — PoE выключен;
- `green` — PoE включён;
- `alternating green-amber` — питание запрещено, потому что будет превышена общая мощность коммутатора;
- `blinking amber` — PoE отключён из-за ошибки;
- `amber` — PoE для порта вручную отключён.

**Главная идея:** по LED-индикаторам можно быстро понять, есть ли питание, линк, передача данных, проблемы с петлями, скорость порта и состояние PoE.

## 1.1.4 Восстановление после сбоя системы

Если IOS не может нормально загрузиться из-за повреждённых или отсутствующих файлов, можно использовать **boot loader**.

Boot loader предоставляет:

- доступ к flash-памяти;
- базовую командную строку;
- возможность указать новый путь к IOS;
- инструменты для восстановления устройства.

### Как войти в boot loader

1. Подключить ПК к console-порту switch через console cable.
2. Настроить terminal emulation software.
3. Отключить питание коммутатора.
4. Снова подключить питание и в течение **15 секунд** нажать и удерживать кнопку **Mode**, пока System LED мигает зелёным.
5. Удерживать кнопку, пока System LED сначала кратко не станет amber, а затем не станет зелёным постоянно.
6. После этого отпустить кнопку Mode.

После успешного входа появится приглашение:

```bash
switch:
```

### Справка по командам boot loader

Чтобы посмотреть доступные команды:

```bash
help
```

или

```bash
?
```

### Просмотр BOOT-переменной

По умолчанию switch пытается загрузиться по пути из переменной **BOOT**.

Для просмотра её значения:

```bash
switch: set
```

Пример:

```bash
switch: set
BOOT=flash:/c2960-lanbasek9-mz.122-55.SE7/c2960-lanbasek9-mz.122-55.SE7.bin
```

### Инициализация flash

Чтобы инициализировать файловую систему flash:

```bash
switch: flash_init
```

Пример вывода:

```bash
switch: flash_init
Initializing Flash...
flashfs[0]: 2 files, 1 directories
flashfs[0]: 0 orphaned files, 0 orphaned directories
flashfs[0]: Total bytes: 32514048
flashfs[0]: Bytes used: 11838464
flashfs[0]: Bytes available: 20675584
flashfs[0]: flashfs fsck took 10 seconds.
...done Initializing Flash.
```

### Просмотр файлов во flash

После инициализации можно посмотреть содержимое flash:

```bash
switch: dir flash:
```

Пример:

```bash
switch: dir flash:
Directory of flash:/
    2  -rwx  11834846  <date>               c2960-lanbasek9-mz.150-2.SE8.bin
    3  -rwx  2072      <date>               multiple-fs
```

### Изменение пути BOOT и запуск IOS

Чтобы указать новый образ IOS:

```bash
switch: BOOT=flash:c2960-lanbasek9-mz.150-2.SE8.bin
```

Проверка:

```bash
switch: set
BOOT=flash:c2960-lanbasek9-mz.150-2.SE8.bin
```

Загрузка IOS:

```bash
switch: boot
```

### Для чего ещё нужен boot loader

Boot loader useful не только для запуска IOS. Он также поддерживает:

- инициализацию flash;
- форматирование flash;
- установку нового IOS;
- изменение переменной BOOT;
- восстановление потерянных или забытых паролей.

## 1.1.5 Доступ к управлению коммутатором

Чтобы коммутатором можно было управлять удалённо, ему нужен **SVI (Switch Virtual Interface)**.

SVI должен иметь:

- либо **IPv4 address + subnet mask**;
- либо **IPv6 address + prefix length**.

### Что такое SVI

SVI — это **виртуальный интерфейс**, а не физический порт коммутатора.

Именно он используется для management access.

### Что ещё нужно для удалённого управления

Если управление будет происходить **из другой сети**, коммутатору также нужен **default gateway**.

Это очень похоже на настройку IP-параметров у обычного хоста.

В рассматриваемом примере используются:

- IPv4 gateway: `172.17.99.1`
- IPv6 network: `2001:db8:acad:99::/64`
- management VLAN: `99`
- IPv4 address коммутатора: `172.17.99.11/24`
- IPv6 address коммутатора: `2001:db8:acad:99::11/64`

---

## 1.1.6 Пример настройки SVI на коммутаторе

По умолчанию switch управляется через **VLAN 1**, и все порты по умолчанию тоже находятся в VLAN 1.

Но с точки зрения безопасности лучшей практикой считается использование **management VLAN, отличного от VLAN 1**.

В примере используется **VLAN 99**.

---

## Шаг 1. Настройка management interface

Из режима настройки VLAN-интерфейса на management SVI коммутатора назначаются:

- IPv4-адрес;
- маска подсети;
- IPv6-адрес.

В примере для **SVI VLAN 99** используются:

- `172.17.99.11/24`
- `2001:db8:acad:99::11/64`

### Важное замечание

SVI для VLAN 99 не будет в состоянии `up/up`, пока:

- VLAN 99 не будет создан;
- и пока к одному из switchport, связанному с VLAN 99, не будет подключено устройство.

### Ещё одно важное замечание

Для IPv6 switch может понадобиться дополнительная настройка.

Например, на **Cisco Catalyst 2960** с **IOS 15.0** перед настройкой IPv6-адресации может потребоваться команда:

```bash
sdm prefer dual-ipv4-and-ipv6 default
```

После этого коммутатор нужно перезагрузить.

### Команды настройки management SVI

```bash
S1# configure terminal
S1(config)# interface vlan 99
S1(config-if)# ip address 172.17.99.11 255.255.255.0
S1(config-if)# ipv6 address 2001:db8:acad:99::11/64
S1(config-if)# no shutdown
S1(config-if)# end
S1# copy running-config startup-config
```

### Что делает каждая команда

- `configure terminal` — вход в глобальный режим конфигурации;
- `interface vlan 99` — переход к настройке SVI для VLAN 99;
- `ip address 172.17.99.11 255.255.255.0` — назначение IPv4-адреса management-интерфейсу;
- `ipv6 address 2001:db8:acad:99::11/64` — назначение IPv6-адреса;
- `no shutdown` — включение интерфейса;
- `end` — возврат в privileged EXEC mode;
- `copy running-config startup-config` — сохранение конфигурации.

---

## Шаг 2. Настройка default gateway

Если коммутатор должен управляться удалённо из другой сети, ему нужен **default gateway**.

### Важное замечание

Для IPv6 в этом примере отдельный default gateway не требуется, потому что switch может получить эту информацию через **RA (Router Advertisement)**.

### Команды настройки шлюза

```bash
S1# configure terminal
S1(config)# ip default-gateway 172.17.99.1
S1(config)# end
S1# copy running-config startup-config
```

### Что делает каждая команда

- `configure terminal` — вход в глобальный режим конфигурации;
- `ip default-gateway 172.17.99.1` — настройка default gateway для коммутатора;
- `end` — возврат в privileged EXEC mode;
- `copy running-config startup-config` — сохранение конфигурации.

---

## Шаг 3. Проверка конфигурации

Для проверки состояния физических и виртуальных интерфейсов удобно использовать команды:

- `show ip interface brief`
- `show ipv6 interface brief`

Пример вывода:

```bash
S1# show ip interface brief
Interface     IP-Address     OK? Method   Status    Protocol
Vlan99        172.17.99.11   YES manual   down      down
```

```bash
S1# show ipv6 interface brief
Vlan99                 [down/down]
    FE80::C27B:BCFF:FEC4:A9C1
    2001:DB8:ACAD:99::11
```

### Что показывает вывод

Вывод подтверждает, что интерфейс **VLAN 99** получил:

- IPv4-адрес;
- IPv6-адрес.

### Важное замечание

IP-адрес на SVI нужен **только для удалённого управления коммутатором**.

Это **не означает**, что switch после этого начнёт маршрутизировать **Layer 3 packets**.

Для обычного L2-коммутатора SVI не превращает устройство в роутер.

---

## Главное, что нужно запомнить

- Switch проходит пять этапов загрузки: `POST -> boot loader -> CPU init -> flash init -> IOS load`.
- Переменная `BOOT` определяет, какой IOS будет загружен.
- LED-индикаторы помогают быстро понять состояние питания, портов, скорости и PoE.
- Через boot loader можно восстановить устройство и вручную указать новый IOS.
- Для удалённого управления switch нужен **SVI**.
- Для управления из другой сети нужен **default gateway**.
- Использовать **VLAN 1** для management не рекомендуется; лучше выделить отдельный management VLAN, например **VLAN 99**.
